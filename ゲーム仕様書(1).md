## 0. この仕様書を読むAIへ

このドキュメントはブラウザで動く1v1ターン制対戦ゲームの完全仕様書である。
以下のことを前提として読むこと。

- バックエンドサーバー・データベースは**一切使わない**
- 対戦はWebRTC（PeerJS）によるP2P通信で行う
- 画像・イラスト等のアセットは**一切ない**。UIはテキストとCSSのみで構築する
- キャラクターデータはJSONファイルで管理し、開発者が手書きする
- 各キャラクターのスキルロジックは、キャラごとに固有のハンドラ関数として実装する
- スキル設計に共通フォーマットは存在しない。キャラによって全く異質な挙動を許容し、むしろ推奨する
- 目標とするスキル複雑度は
 Companyレベル（複数条件・リソース管理・状態依存の多重分岐）

## 1. ゲームコンセプト

### 一言で言うと
「キャラクター1体がカードデッキのような複雑さを持つ、ターン制1v1対戦ゲーム」

### 設計思想
- 収集・育成・課金要素は**一切なし**。全キャラ最初から使用可能。
- 「このキャラを極める」習熟の深さが長期的な楽しさの核心。
- スキルはカテゴリ分類しない。キャラごとに全く異質な設計を許容する。
- 1キャラ内のスキルシナジー × 編成間のシナジーという二層の戦略性。
- さくっと始められて、やればやるほど深くなる。

## 2. コアゲームプレイ

### 2.1 対戦形式
- 1v1 同期対戦（P2P・リアルタイム）
- 先攻・後攻はコイントス（完全ランダム）
- **ポケモン形式：場に出ているのは常に1体のみ。控えの2体は同時出場しない。**
- **勝利条件: 相手チーム3体全滅のみ**

### 2.2 編成・交代ルール
- 各プレイヤーは3体のキャラクターでチームを組む（試合前に確定）
- 全キャラクターは最初から選択可能
- **試合開始時に先頭に出すキャラ1体だけを選択する**
- 残り2体の出順は事前に決めない
- **交代は原則スキル使用前のみ可能**
  - 例外: キャラ固有スキルによって「スキル使用後に交代可能」「スキル効果として交代を行う」場合がある
  - 例外はキャラ設計の中で明示する
- 交代自体にコストはかからないが、交代後そのターンの残コストは消滅しターンが終了する
- **死亡時は控えの2体からその場で選んで出す（30秒以内。時間切れはランダム選出）**

### 2.3 ターン構造

```
【ターン開始】
  ↓
コスト3を付与（一部スキル効果で増減する場合あり）
  ↓
【行動フェーズ】以下を任意の順で実行
  ├─ 交代する（スキル使用前のみ・交代後はターン終了）※キャラ固有の例外あり
  └─ スキルを使用する（コストの範囲内で複数使用可能）
       例: コスト1×3 / コスト2+コスト1 / コスト3×1 など
       残コストは翌ターンに持ち越さず消滅
  ↓
【効果処理】スキル効果を順次処理
  ↓
【敵撃破判定】
  HPが0以下の敵キャラを戦闘不能にする
  ├─ 敵を倒した場合 → 攻撃側のターンはその時点で終了
  │    敵チームに控えがいれば、相手プレイヤーが次キャラを選択（30秒以内・時間切れはランダム）
  │    選択完了後、相手（新キャラを出した側）のターンが始まる
  └─ 敵を倒していない場合 → 行動フェーズを継続
  ↓
【ターン終了処理】（プレイヤーが「ターン終了」を選択、またはコスト消費後）
  - DoT（継続ダメージ）の適用
  - リジェネ（継続回復）の適用
  - バフ・デバフの持続ターン減少
  - 派生スキルの解放状態の更新
  ↓
【死亡判定】
  両チームの生存確認 → 全滅していれば試合終了
  ↓
【ターン交代】相手のターンへ
```

### 2.4 スキルシステム

#### 基本スキル
- 各キャラが持つスキル数は**自由**（1種類だけのキャラも、10種類以上のキャラも可）
- カテゴリ分類（攻撃・補助・回復など）は**しない**
- キャラごとに効果・挙動・条件は完全に異なる
- 全スキル共通で持つのは「コスト（1〜3の整数）」と「説明文」のみ

#### 派生スキル（キャラによって有無・深さが異なる）
- 特定のスキルを使用すると、追加のスキルが解放される場合がある
- **解放されたスキルは基本的に同ターン内で使用可能**（キャラによって異なる）
- 派生スキルをさらに使うと次の派生スキルが解放される「チェーン」も可
- 翌ターンへ持ち越す形の派生もキャラ設計次第で可能
- 派生スキルの有無・深さ・持続はキャラごとに完全自由

#### スキルシナジー（キャラ内）
- 同一ターン内で特定の複数スキルを組み合わせると追加効果が発動する場合がある
- これもキャラごとに独自設計。共通ルールなし

### 2.5 パッシブ能力
- パッシブは**先頭に出ているキャラだけでなく、控えから発動するものも存在する**
- 発動タイミング・条件はキャラごとに完全自由
- 例: 「控えに味方が死亡した時に発動」「編成に特定属性がいる間常時発動」など

#### パッシブトリガー一覧

バトルエンジンは以下のタイミングでパッシブを呼び出す。
キャラを追加するたびに必要なトリガーを追記していく（随時拡張）。

| トリガーID | 発火タイミング |
|---|---|
| `on_turn_start` | 自分のターン開始時 |
| `on_turn_end` | 自分のターン終了時 |
| `on_skill_used` | 自分がスキルを使用した後 |
| `on_enemy_skill_used` | 相手がスキルを使用した後 |
| `on_damage_received` | 自分がダメージを受けた時 |
| `on_ally_death` | 味方が死亡した時（控えからでも発動） |
| `on_enemy_death` | 相手が死亡した時 |
| `on_switch_in` | 自分が場に出た時 |
| `on_switch_out` | 自分が控えに下がった時 |
| `on_ally_switch` | 味方が交代した時 |
| `on_cost_gained` | コストが付与された時 |
| `passive_always` | 常時発動（編成にいるだけで効果） |
| `passive_while_active` | 場に出ている間のみ常時発動 |
| `passive_while_bench` | 控えにいる間のみ常時発動 |
| `passive_while_buff` | 特定バフ保持中のみ常時発動 |
| `passive_while_resource_above` | 固有リソースが閾値以上の間のみ常時発動 |

### 2.6 優先度システム
- 通常、先攻・後攻はコイントスで決まったターン交代制で固定される
- ただし、バフやスキル効果によって**ターン開始時の優先度が変わる**場合がある
- 優先度が高い側はそのターン内で相手より先に処理される
- 優先度の発動条件・効果はキャラ固有設計による

### 2.7 編成シナジー
- 特定の属性の組み合わせや、特定のパッシブの組み合わせでボーナスが発生する場合がある
- 共通ルールは設けず、キャラ設計の中で自然に発生させる
- **ペアシナジー・トリオシナジー**：特定の2体または3体が同じ編成にいる場合に発動する特殊パッシブを設計できる
  - 例: ペアの片方が死亡したとき、もう片方が覚醒して別のスキルセットに変化
  - 例: トリオ全員生存中、専用スキルが出現する
  - 実装はキャラ固有パッシブ（`passive_always`など）として設計する

## 3. キャラクター設計

### 3.1 基本ステータス
| ステータス | 説明 |
|---|---|
| `hp` | 最大HP |
| `atk` | 攻撃力（ダメージ計算に使用） |
| `def` | 防御力（被ダメージ軽減） |
| `spd` | 速度（将来用途のために保持） |

### 3.2 1キャラあたりの構成
```
スキル × 1〜n（キャラによって完全に異なる・最低1種）
派生スキル × 0〜n（任意・キャラによる）
パッシブ × 0〜n（任意・キャラによる）
固有リソース × 0〜n（任意・キャラによる）
```

### 3.3 キャラ固有リソース（任意）
- キャラによっては独自のリソース（スタック・ゲージ・カウンターなど）を持つ場合がある
- 例: 「怨恨スタック」「充填ゲージ」「共鳴値」など
- これらはキャラ固有のステートとしてバトルエンジンが管理する
- JSONの`custom_resources`フィールドで定義する

### 3.4 キャラ数
- リリース時点: **10体**
- 追加はリリース後に随時JSONとハンドラをセットで追加する

|---|---|
| 🔥 火 (fire) | 🌿 草 (grass) |
| 💧 水 (water) | 🔥 火 (fire) |

| ⬜ 無 (none) | 相性補正なし |

- 有利属性へのダメージ: **×1.2**
- 不利・中立・無属性への補正: なし

## 5. ダメージ計算

基本式は以下の通りだが、**バフ・デバフ・スキル固有の補正・状態異常・キャラ固有リソースなど
多数の要素が絡むため、最終的な計算はキャラ固有ハンドラ内で柔軟に処理する**。

```
基本ダメージ = (攻撃側ATK × スキル倍率) - (防御側DEF / 2)
最終ダメージ = 基本ダメージ × 属性補正 × 各種バフ・デバフ補正 × スキル固有補正 ...
最小ダメージ = 1（0以下にはならない）
```

- スキル倍率・固有補正はハンドラ内で定義する
- バフ・デバフは`buffs`・`debuffs`配列で管理し、ハンドラから参照する
- ダメージ計算のユーティリティ関数（`damage.ts`）を用意するが、あくまで補助であり、
  ハンドラが自由にオーバーライドして良い

## 6. 技術スタック

### 6.1 全体方針
- **サーバーレス・DBレス**
- 対戦データはP2P（WebRTC）で直接やり取りする
- PeerJSの無料シグナリングサーバーを使って接続を確立する
- ゲームのロジックはどちらか一方（ホスト側）のブラウザで処理する

### 6.2 使用技術
| 項目 | 技術 | 理由 |
|---|---|---|
| フレームワーク | Next.js (App Router) | 現在の標準・AIの知識量が多い |
| 言語 | TypeScript | 型安全・バグを減らしやすい |
| スタイリング | Tailwind CSS | AIが扱いやすい |
| 状態管理 | Zustand | シンプル |
| P2P通信 | PeerJS | WebRTCのラッパー。無料シグナリングサーバー付き |
| ホスティング | Vercel | 無料・Next.jsと相性が良い |

### 6.3 P2P通信の仕組み

```
【接続確立フロー】
プレイヤーAがPeerJSでPeer IDを取得
  ↓
AがルームコードとしてPeer IDを相手に共有（URLやチャット経由）
  ↓
プレイヤーBがPeer IDを入力してAに接続
  ↓
WebRTC接続確立（以降はサーバーを経由しない）
  ↓
【対戦中】
ホスト（A）がゲーム状態を管理・バトルロジックを処理
プレイヤーのアクション（スキル選択・交代）をP2Pで送受信
ホストがロジックを処理し、結果を両者に即時送信（極力リアルタイムに反映）
```

### 6.4 注意事項
- ゲームロジックはホスト側で処理するため、ホストが有利な情報を持つ
- 内輪（知人間）での利用を前提としているため、チート対策は現時点では不要
- 将来的にサーバーを追加する場合は、バトルエンジンをサーバーに移行する

## 8. バトルエンジン設計

### 8.1 基本方針
- バトルロジックはホスト側のブラウザで処理する
- ゲスト側はアクションを送信し、結果を受け取るだけ
- ファイル: `src/lib/battle-engine.ts`

### 8.2 ゲーム状態の型定義

```typescript
type Element = 'fire' | 'water' | 'grass' | 'light' | 'dark' | 'none';

type CustomResource = {
  id: string;
  name: string;
  value: number;
  min: number;
  max: number;
};

type Buff = {
  id: string;
  stat: string;
  multiplier: number;
  turnsRemaining: number;
};

type Debuff = {
  id: string;
  stat: string;
  multiplier: number;
  turnsRemaining: number;
};

type DoT = {
  id: string;
  damagePerTurn: number;
  turnsRemaining: number;
};

type Regen = {
  id: string;
  healPerTurn: number;
  turnsRemaining: number;
};

type CharacterState = {
  id: string;
  name: string;
  element: Element;
  hp: number;
  maxHp: number;
  atk: number;
  def: number;
  spd: number;
  customResources: CustomResource[];
  buffs: Buff[];
  debuffs: Debuff[];
  dots: DoT[];
  regens: Regen[];
  unlockedDerivedSkillIds: string[];   // 現在このターン解放中の派生スキルID
  nextTurnUnlockedSkillIds: string[];  // 次ターンに解放される派生スキルID
  priority: number;                    // 優先度（デフォルト0・高いほど先行）
  isAlive: boolean;
  isActive: boolean;
};

type TeamState = {
  playerId: string;
  activeIndex: number;
  characters: CharacterState[];
};

type BattleState = {
  team1: TeamState;
  team2: TeamState;
  turn: number;
  currentTurn: 'team1' | 'team2';
  phase: 'waiting' | 'selecting_first' | 'action' | 'selecting_next' | 'finished';
  remainingCost: number;
  winner: 'team1' | 'team2' | null;
  log: BattleLog[];
};

type PlayerAction =
  | { type: 'select_first'; characterIndex: number }
  | { type: 'use_skill'; skillId: string }
  | { type: 'switch_character'; characterIndex: number }
  | { type: 'end_turn' }
  | { type: 'select_next'; characterIndex: number };

type BattleLog = {
  turn: number;
  actor: 'team1' | 'team2';
  description: string;
  stateSnapshot: BattleState;
};
```

> `BattleLog`はフルスナップショットを持つためメモリを消費しやすい。
> 将来的に「直近N件のみ保持」または「差分記録」への変更を検討すること。

### 8.3 スキルハンドラの実装方針

各キャラクターの固有スキルは`src/lib/skill-handlers/`以下にキャラごとのファイルで実装する。

```typescript
// src/lib/skill-handlers/char_001.ts
type SkillHandler = (state: BattleState, actorTeam: 'team1' | 'team2') => BattleState;
type PassiveHandler = (state: BattleState, ownerTeam: 'team1' | 'team2') => BattleState;

export const char_001_skill_handlers: Record<string, SkillHandler> = {
  'char_001_skill_1': (state, actorTeam) => {
    // stateを直接変更せず、新しいstateを返す（イミュータブル）
    // JSONのdescriptionに書かれた効果をここに実装する
    return newState;
  },
};

export const char_001_passive_handlers: Record<string, PassiveHandler> = {
  'char_001_passive_1': (state, ownerTeam) => {
    // 控えからでも発動するパッシブの実装
    return newState;
  },
};
```

### 8.4 ターン処理フロー（コード構造）

```typescript
function processAction(state: BattleState, action: PlayerAction): BattleState {
  switch (action.type) {

    case 'select_first':
      return processSelectFirst(state, action.characterIndex);

    case 'use_skill': {
      const skill = findSkill(state, action.skillId);
      if (skill.cost > state.remainingCost) throw new Error('コスト不足');

      let newState = getSkillHandler(action.skillId)(state, state.currentTurn);
      newState = { ...newState, remainingCost: newState.remainingCost - skill.cost };
      newState = processUnlocks(newState, action.skillId, state.currentTurn);
      newState = processDeathCheck(newState);

      // 敵を倒した場合はターン終了処理へ（select_nextフェーズへ移行）
      if (newState.phase === 'selecting_next') return newState;

      newState = triggerPassives(newState, 'on_skill_used');
      return newState;
    }

    case 'switch_character':
      // スキル使用後の交代は不可（キャラ固有例外を除く）
      return processSwitchAndEndTurn(state, action.characterIndex);

    case 'end_turn':
      return processEndTurn(state);

    case 'select_next':
      // 敵撃破後の次キャラ選択（相手プレイヤーが行う）
      // 選択完了後、選択した側のターンが始まる
      return processSelectNext(state, action.characterIndex);
  }
}

function processEndTurn(state: BattleState): BattleState {
  let s = state;
  s = applyDoTs(s);
  s = applyRegens(s);
  s = decrementBuffs(s);
  s = promoteNextTurnUnlocks(s);
  s = clearSameTurnUnlocks(s);
  s = processDeathCheck(s);
  s = processWinCheck(s);
  s = triggerPassives(s, 'on_turn_end');
  s = switchTurn(s);
  return s;
}
```

### 8.5 CPUロジック（ソロモード用）
- CPUはランダムにスキルを選択して使用する
- コストが0になるかランダムなタイミングでターン終了する
- 死亡後の次キャラ選択もランダム
- チート対策・戦略AIは実装しない

## 9. UI設計方針

### 9.1 基本方針
- 画像・イラストは一切使わない
- テキストとCSSのみで構築する
- キャラクターは属性シンボル（絵文字）＋名前で表現する
- **UIデザインは今後繰り返し改良していく前提で、初期実装はシンプルさ優先**
- 最終的には高い情報密度を目指す（複数リソース・状態・スキル情報が同時に把握できる画面）

### 9.2 情報公開ルール
対戦中、以下の情報はすべて相手にも公開される（隠し要素なし）：
- 相手の残コスト
- 相手のバフ・デバフ
- 相手の固有リソース
- 控えキャラの状態異常・バフ・固有リソース（何かついている場合）
- 相手のHPバー（3体分）

### 9.3 画面構成

```
【ログイン・ホーム画面】
  - プレイヤー名の設定・編集
  - 称号の設定・編集（将来的に追加）
  - 「ルームを作る」ボタン → ルームコード（Peer ID）が表示される
  - 「ルームに参加する」入力欄 → ルームコードを入力して接続
  - 「ソロモード（CPU戦）」ボタン

【ロビー画面】（接続後・両者が入室した状態）
  - キャラクター一覧（名前・属性・ステータス・スキル説明を閲覧可能）
  - 3体を選択して編成確定
  - 相手の編成確定状態を表示（確定したかどうかのみ。編成内容は非公開）
  - 両者が確定ボタンを押すと先頭選択へ進む
  - 編成確定後・相手が確定する前であれば変更可能

【先頭選択画面】
  - 自分の3体が表示される
  - 15秒以内に先頭に出すキャラを選択（時間切れはランダム）
  - 両者が選択完了後、同時にバトル開始
  - 相手が何を選んだかは非公開（同時公開）

【バトル画面】
  ┌──────────────────────────────────────┐
  │  相手チーム（HPバー×3・状態表示）   │
  │  相手の先頭キャラ詳細                │
  │  （HP・バフ・デバフ・固有リソース・残コスト） │
  ├──────────────────────────────────────┤
  │  バトルログ（直近の行動をリアルタイム表示） │
  ├──────────────────────────────────────┤
  │  自分の先頭キャラ詳細                │
  │  （HP・バフ・デバフ・固有リソース）  │
  │  自分チーム（HPバー×3・状態表示）   │
  │  スキルボタン（コスト・説明表示）    │
  │  残コスト表示                        │
  │  交代ボタン（スキル未使用時のみ有効）/ ターン終了ボタン │
  └──────────────────────────────────────┘

【死亡後 次キャラ選択画面】
  - 生存している控えを表示・選択
  - 30秒以内に選択（時間切れはランダム）

【リザルト画面】
  - 勝敗表示
  - 「もう一度」ボタン
```

### 9.4 スキルボタンの表示
- コストと名前を常時表示
- 使用可能（残コスト以下）: 通常表示
- 使用不可（残コスト不足）: グレーアウト
- 派生スキルは解放時のみスキルリストに追加表示
- ホバーまたはタップで詳細説明（description）をツールチップ表示

### 9.5 リアルタイム同期方針
- スキル使用・交代などのアクションはP2P経由で即時送信し、極力リアルタイムに相手画面へ反映する
- ホストがロジックを処理して結果を送信するため若干のラグは許容範囲

```
src/
  ├─ app/
  │   ├─ page.tsx                 # ログイン・ホーム画面
  │   ├─ lobby/page.tsx           # ロビー画面（ルーム作成・参加・編成）
  │   ├─ battle/page.tsx          # バトル画面
  │   └─ solo/page.tsx            # ソロモード（CPU戦）
  ├─ components/
  │   ├─ CharacterCard.tsx        # キャラ情報表示
  │   ├─ SkillButton.tsx          # スキルボタン
  │   ├─ HPBar.tsx                # HPバー
  │   ├─ ResourceDisplay.tsx      # 固有リソース表示
  │   ├─ StatusEffects.tsx        # バフ・デバフ表示
  │   └─ BattleLog.tsx            # ログ表示
  ├─ lib/
  │   ├─ battle-engine.ts         # バトルロジック本体・ターン処理
  │   ├─ p2p.ts                   # PeerJS接続管理
  │   ├─ damage.ts                # ダメージ計算ユーティリティ
  │   ├─ passives.ts              # パッシブトリガー管理
  │   ├─ cpu.ts                   # CPUロジック（ソロモード用）
  │   └─ skill-handlers/
  │       ├─ index.ts             # 全ハンドラのエクスポート
  │       ├─ char_001.ts
  │       └─ ...（キャラ数分）
  ├─ data/
  │   └─ characters.json          # キャラクターマスターデータ
  ├─ store/
  │   └─ gameStore.ts             # Zustand ストア
  └─ types/
      └─ index.ts                 # 型定義
```

## 11. 実装の優先順位

1. **型定義** (`types/index.ts`)
2. **キャラクターデータの雛形** (`characters.json` 1〜2体分)
3. **バトルエンジンのコア** (`battle-engine.ts`, `damage.ts`)
4. **パッシブ管理** (`passives.ts`)
5. **スキルハンドラの雛形** (`skill-handlers/`)
6. **CPUロジック** (`cpu.ts`)
7. **P2P接続** (`p2p.ts`)
8. **バトル画面UI** (`battle/page.tsx` + 各コンポーネント)
9. **ロビー画面・編成** (`lobby/page.tsx`)
10. **ホーム・ソロモード画面** (`page.tsx`, `solo/page.tsx`)

## 12. 開発フロー（人間とAIの役割分担）

| 作業 | 担当 |
|---|---|
| キャラのコンセプト・スキル内容を考える | 人間（稀にAI） |
| `characters.json`にキャラデータを記述する | AI（場合によっては人間が補助） |
| JSONの`description`を読んでハンドラを実装する | AI |
| バトルエンジン・P2P・UI全般の実装 | AI |
| バランス調整（JSONの数値変更） | 人間 |
| 新キャラ追加（JSON記述 → ハンドラ生成） | 人間 + AI |
| UIの改良・デザイン更新 | AI（人間の指示ベース） |
